# 2. Connect Token System

## Overview

Connect tokens are the core authentication mechanism. They are generated by the web backend (not the game server) and given to authenticated clients. The server validates them without contacting the backend.

## Connect Token Structure (2048 bytes total)

### Public Portion (readable by client)

```
[version info]           13 bytes    "NETCODE 1.02\0"
[protocol id]            8 bytes     uint64 — unique per game
[create timestamp]       8 bytes     uint64 — unix timestamp
[expire timestamp]       8 bytes     uint64 — unix timestamp
[connect token nonce]    24 bytes    random, used for decryption
[encrypted private data] 1024 bytes  see below
[timeout seconds]        4 bytes     uint32
[num server addresses]   4 bytes     uint32 (1-32)
[server addresses]       variable    up to 32 addresses
[client to server key]   32 bytes    ChaCha20 key
[server to client key]   32 bytes    ChaCha20 key
<zero pad to 2048 bytes>
```

### Private Portion (encrypted, only backend + server can read)

```
[client id]              8 bytes     uint64 — globally unique
[timeout seconds]        4 bytes     uint32
[num server addresses]   4 bytes     uint32
[server addresses]       variable    up to 32 addresses
[client to server key]   32 bytes    ChaCha20 key
[server to client key]   32 bytes    ChaCha20 key
[user data]              256 bytes   application-defined
<zero pad to 1024 bytes>
```

Private portion encrypted with **XChaCha20-Poly1305** using:
- Key: shared private key (32 bytes, known by backend + server only)
- Nonce: 24-byte random value (stored in public portion)
- AAD: version info + protocol id + expire timestamp

## Security Properties

1. **Cannot be forged** — encrypted with private key only backend and server know
2. **Cannot be modified** — AEAD authentication detects tampering
3. **Expires** — timestamp-based expiration prevents replay
4. **IP-independent** — not bound to specific client IP (supports NAT, migration)
5. **Server list** — client can try multiple servers if first is full/down
6. **User data** — 256 bytes of application-defined data (player ID, matchmaking info, etc.)
7. **Separate keys** — each direction has its own encryption key

## Generation API

```c
int netcode_generate_connect_token(
    int num_server_addresses,
    const char ** public_server_addresses,    // addresses client sees
    const char ** internal_server_addresses,  // addresses server checks internally
    int expire_seconds,
    int timeout_seconds,
    uint64_t client_id,
    uint64_t protocol_id,
    const uint8_t * private_key,             // 32-byte shared secret
    uint8_t * user_data,                     // 256 bytes application data
    uint8_t * connect_token                  // output: 2048 bytes
);
```

## Key Insight for netudp

The connect token system is **the** gold standard for UDP game server auth. It solves:
- Authentication without per-server databases
- Anti-DDoS (server allocates nothing until valid token received)
- Matchmaking integration (backend assigns servers)
- Separate encryption keys per direction

netudp should implement this exact pattern.
