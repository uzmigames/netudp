# 5. Encryption (libsodium AEAD)

## Algorithms Used

| Purpose | Algorithm | Library Function |
|---|---|---|
| Connect token encryption | XChaCha20-Poly1305 | `crypto_aead_xchacha20poly1305_ietf_encrypt` |
| Packet encryption | ChaCha20-Poly1305 (IETF) | `crypto_aead_chacha20poly1305_ietf_encrypt` |
| Challenge token | ChaCha20-Poly1305 (IETF) | `crypto_aead_chacha20poly1305_ietf_encrypt` |
| Random bytes | OS CSPRNG | `randombytes_buf` |

## Why Two Different Ciphers?

- **XChaCha20** (24-byte nonce) for connect tokens — random nonce per token, no sequence tracking
- **ChaCha20** (12-byte nonce) for packets — sequence number as nonce, deterministic

## Key Management

```
Connect Token contains:
  [client_to_server_key]  32 bytes  — generated by backend
  [server_to_client_key]  32 bytes  — generated by backend

Packet encryption:
  Client → Server: encrypted with client_to_server_key, nonce = packet sequence
  Server → Client: encrypted with server_to_client_key, nonce = packet sequence

Challenge token:
  Encrypted with server's random key (generated at server start, never shared)
```

## Nonce Construction

```c
// Packet nonce: sequence number zero-padded to 96 bits (12 bytes)
uint8_t nonce[12];
memset(nonce, 0, sizeof(nonce));
// write sequence as little-endian uint64 into first 8 bytes
```

Since sequence numbers are monotonically increasing and never reused, nonces are guaranteed unique per key.

## libsodium Dependency

netcode.io bundles a vendored subset of libsodium (the `sodium/` directory) containing only the required primitives. This avoids requiring a system-wide libsodium installation.

For netudp, we should similarly vendor or bundle the crypto primitives rather than requiring an external dependency.
