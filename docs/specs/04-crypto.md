# Spec 04 — Cryptography

## Requirements

### REQ-04.1: AEAD — ChaCha20-Poly1305

Primary cipher for all packet encryption after handshake.

```cpp
namespace netudp::crypto {

// Encrypt plaintext with associated data
// Returns ciphertext_len = plaintext_len + 16 (Poly1305 tag)
int aead_encrypt(
    const uint8_t key[32],
    const uint8_t nonce[12],
    const uint8_t* aad, int aad_len,
    const uint8_t* plaintext, int plaintext_len,
    uint8_t* ciphertext  // Must have capacity: plaintext_len + 16
);

// Decrypt and verify
// Returns plaintext_len or -1 on auth failure
int aead_decrypt(
    const uint8_t key[32],
    const uint8_t nonce[12],
    const uint8_t* aad, int aad_len,
    const uint8_t* ciphertext, int ciphertext_len,
    uint8_t* plaintext  // Must have capacity: ciphertext_len - 16
);

}
```

### REQ-04.2: XChaCha20-Poly1305

For connect token encryption (24-byte random nonce).

```cpp
int xchacha_encrypt(const uint8_t key[32], const uint8_t nonce[24],

```

### REQ-04.3: Nonce Construction

```cpp
// Packet nonce: sequence zero-padded to 12 bytes
void build_nonce(uint64_t sequence, uint8_t nonce[12]) {
    memset(nonce, 0, 12);
    memcpy(nonce, &sequence, 8);  // Little-endian
}
```

Nonce is deterministic. Never reused because sequence is monotonically increasing per key.

### REQ-04.4: Key Management
- **Tx key** (32 bytes): client→server direction. From connect token.
- **Rx key** (32 bytes): server→client direction. From connect token.
- Keys are separate per direction (prevents reflection attacks).
- Keys are generated by the web backend and embedded in the connect token.
- No online key exchange (cheaper than ECDH).

### REQ-04.5: CRC32C

Hardware-accelerated checksum for LAN/dev mode and internal integrity.

```cpp
uint32_t crc32c(const uint8_t* data, int len);
// SIMD dispatch: SSE4.2 _mm_crc32_u64, ARM __crc32cd, or software table
```

### REQ-04.6: CSPRNG

```cpp
void random_bytes(uint8_t* data, int len);
// Platform: CryptGenRandom (Windows), /dev/urandom (Unix), arc4random (macOS)
```

### REQ-04.7: Automatic Rekeying

When `bytes_transmitted >= 1GB` OR `session_duration >= 1 hour`:
1. Derive new keys: `HKDF-SHA256(old_key, "netudp-rekey" || max_seq)`
2. Reset sequence counters to 0
3. Reset replay protection window
4. Both sides derive independently (deterministic from shared state)

### REQ-04.8: AAD (Associated Data)

```
[version_info]  13 bytes   "NETUDP 1.00\0"
[protocol_id]   8 bytes    uint64 LE
[prefix_byte]   1 byte     packet type + seq bytes
```

Total: 22 bytes. Authenticated but not encrypted.

### REQ-04.9: Replay Protection

```cpp
struct ReplayProtection {
    uint64_t most_recent;
    uint64_t received[256];  // received[seq % 256] = seq

    bool check(uint64_t seq);    // True if already received or too old
    void advance(uint64_t seq);  // Mark as received, update most_recent
    void reset();
};
```

Packets with `seq + 256 <= most_recent` are too old → rejected.
Packets with `received[seq % 256] == seq` are duplicates → rejected.

## Scenarios

#### Scenario: Encrypt and decrypt round-trip
Given key, nonce, plaintext "Hello"
When encrypt then decrypt
Then output matches "Hello" exactly

#### Scenario: Tampered ciphertext
Given valid ciphertext
When 1 bit is flipped in the ciphertext
Then decrypt returns -1 (auth failure)

#### Scenario: Replay rejection
Given packet with sequence 100 already received
When the same packet arrives again
Then `replay.check(100)` returns true (reject)

#### Scenario: Automatic rekey
Given 1GB of data transmitted
When `should_rekey()` is checked
Then returns true
When `perform_rekey()` is called
Then new keys derived, sequences reset, old keys zeroed
