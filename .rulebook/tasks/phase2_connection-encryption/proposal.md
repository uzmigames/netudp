# Proposal: Connection + Encryption — Crypto, Connect Tokens, Handshake

## Why
Game servers require encrypted authenticated connections to prevent cheating, packet injection, and DDoS amplification. The netcode.io protocol provides a proven model: connect tokens generated by a backend, a 4-step handshake, and per-packet AEAD encryption with replay protection. This phase delivers the entire connection lifecycle: token generation → client request → server challenge → client response → encrypted bidirectional data exchange. Without this, all traffic is plaintext and unauthenticated.

## What Changes
- Vendored ChaCha20-Poly1305 (monocypher) + XChaCha20 for tokens + CRC32C (generic+SIMD) + CSPRNG
- Connect token structure (2048 bytes): generation, encryption, server-side validation, fingerprint anti-replay
- AEAD packet encryption: 64-bit nonce counter per key epoch, AAD with version+protocol+prefix
- Replay protection: 256-entry window on 64-bit nonce counter
- 4-step handshake: CONNECTION_REQUEST → CHALLENGE → RESPONSE → KEEPALIVE (connected)
- Client state machine: 10 states (6 error + 4 normal), multi-server fallback
- Per-IP token bucket rate limiting (60/s, burst 10)

## Impact
- Affected specs: 04-crypto, 05-connect-tokens, 09-wire-format (CONNECTION_REQUEST layout)
- Affected code: src/crypto/, src/connection/, include/netudp/netudp_token.h
- Breaking change: NO (greenfield)
- User benefit: Secure authenticated connections with zero online key exchange
